# Command Center — Gap Audit (2026-07-08)

State audited: main @ 9c32e0f (v1+v2+v2.2+v2.3 deployed, beta-gated). 3-lens workflow audit (deferred inventory, operator journey, production reality) + opus synthesis.


## 1. Para usarlo HOY

### Encender el flag y armar dry-run (COMMAND_CENTER_BETA + CC_DRY_RUN) `[blocker-to-use]`
Hoy todo el Command Center es invisible: access.ts exige COMMAND_CENTER_BETA==='true' antes de cualquier check y la var no existe en Coolify (app tk8s8s4k44s8co8ow8cw8koo), asi que cada /command y /api/command da 404. Paso 1: setear COMMAND_CENTER_BETA=true y, en el mismo cambio, CC_DRY_RUN=true para que el primer Ejecutar sea un validateOnly y no mute cuentas reales, luego reiniciar. Sin CC_DRY_RUN, en cuanto enciendes el beta, Ejecutar hace mutaciones reales.

*Evidencia:* src/lib/command/access.ts:15-20; DEPLOY-NOTES-command-center.md Release steps 2; live Coolify env (ambas vars ausentes)

### Correr migraciones 007/008/009 (POST /api/migrate como admin) `[blocker-to-use]`
007 (cc_actions/cc_executions/cc_settings), 008 (cc_blueprints + columnas + allow-list create_*) y 009 (create_adset) pueden no estar aplicadas en prod; no verificable desde aqui porque ADS_DATABASE_URL apunta a host interno de Coolify. Sin 008, cada publish de /command/crear y approve de /command/editar falla (no existe cc_blueprints). Accion: loguearte como admin en ads.airankia.com, POST /api/migrate, y confirmar que schema_migrations tiene 007/008/009 y que cc_settings.allowed_action_types incluye create_campaign y create_adset.

*Evidencia:* src/app/api/migrate/route.ts:592-627; DEPLOY-NOTES-command-center.md Migration; live Coolify ADS_DATABASE_URL

### Conectar una cuenta Google en Conexiones (con conversion action ENABLED) `[blocker-to-use]`
Todo path Google es connection-scoped: blueprint POST rechaza sin connection_id del workspace y networks/google.ts solo emite tokens desde el refresh token de la conexion (nunca usa GOOGLE_ADS_REFRESH_TOKEN de env). El flujo de connect ya esta listo (OAuth client + CONNECTIONS_KEY presentes en Coolify). Accion: en el primer login ir a Conexiones y conectar la cuenta del workspace; para poder crear, esa cuenta necesita una conversion action ENABLED o los blueprints con automated bidding fallan en validateOnly.

*Evidencia:* src/app/api/command/blueprint/route.ts:57-67; src/lib/command/networks/google.ts:1-27; DEPLOY-NOTES Preconditions to actually publish

### Ensayo: dry-run por accion, un budget_update real + Revertir, luego quitar CC_DRY_RUN `[blocker-to-use]`
El protocolo de release es una semana con CC_DRY_RUN=true rehearsando acciones v1 sueltas, luego un budget_update real seguido de Revertir para probar ledger/rollback, y solo tras una semana limpia en Bitacora quitar CC_DRY_RUN. Ojo: dry-run solo rehearsa acciones sueltas — los blueprints multi-accion fallan cerrados en el primer tmp: ref, asi que el flujo create se ensaya aparte con una campana PAUSED desechable en una cuenta real. Es seguro (falla cerrado) pero debilita el ensayo del create/edit.

*Evidencia:* DEPLOY-NOTES-command-center.md Release steps 4-6 + CC_DRY_RUN and multi-action blueprints


## 2. Huecos de alto valor

### Sin datos de performance en el loop de operacion (M) `[high-value-gap]`
La tabla de campanas del Command Center muestra solo status, presupuesto diario y learning phase (CampaignRow no trae spend/clicks/conversiones/CPA), asi que el operador decide que pausar o re-presupuestar sin evidencia enfrente. Google performance vive solo en /performance (sentinel externo, Google-only) y Meta no tiene reporte de performance en ningun lado (insights solo se usan dentro de snapshot() para gates). Cerrarlo = columnas de metricas + rangos de fecha en cuentas/acciones para ambas redes; los adapters ya saben consultar insights/GAQL. Desbloquea decisiones basadas en evidencia sin salir a las UIs nativas.

*Evidencia:* src/app/command/cuentas/cuentas-client.tsx:14-21; src/app/performance/page.tsx:46; src/lib/command/networks/meta.ts:265-269

### Aprobar/ejecutar de a una fila y import del motor con IDs a mano (S) `[high-value-gap]`
AccionesClient solo expone Aprobar/Ejecutar por fila (call() toma un id), asi que procesar un lote de recomendaciones del motor son N clicks x N confirmaciones semanales. El form 'Importar del motor' exige teclear engine_account_id, connection_id y customer_id en tres inputs libres — datos que la app ya tiene y podria ofrecer en un picker. Seleccion multiple + selector pre-poblado son cambios chicos sobre los endpoints existentes; alto alivio semanal por poco costo.

*Evidencia:* src/app/command/acciones/acciones-client.tsx:41-62, :89-101

### Edit mode no hace higiene semanal (M/L) `[high-value-gap]`
La superficie editable es exactamente status+budget de campana, newNegatives (solo alta), status de ad-group, newKeywords (solo alta), reemplazo de ad y newAds; baseKeywords son server-owned sin flag de remocion, no hay campo desired para nombres, ni nada para bidding/geo/schedule/crear ad group; ads no-RSA se marcan unsupported y campanas no-SEARCH se rechazan con 409. Podar search terms y ajustar pujas es el core del loop de optimizacion, asi que el operador sigue viviendo en la UI de Google. El rail YA anuncia remove_negatives/remove_entity — el edit doc simplemente nunca los emite; falta un verb user-facing de remove + una familia update_* con sus propias gate stories.

*Evidencia:* src/lib/command/edit/schema.ts:38-77; read-tree.ts:49; edit route:84-86; src/lib/command/networks/google.ts:316-319; v2.3 design §h

### Sin notificaciones ni verificacion post-ejecucion (M) `[high-value-gap]`
No hay codigo de notificacion en src/lib/command ni /api/command (ni email, webhook, ni alerta in-app): si una ejecucion falla a media plan, un gate bloquea o hace falta rollback, nada avisa. Ademas el state machine define 'verified' y 'expired' pero ningun codigo transiciona a ellos — no hay drift-check que confirme que el cambio pego, y las acciones aprobadas nunca expiran aunque la campana haya cambiado desde el approve. Un cron de verificacion + un canal simple (email/Telegram) lo cierra y da confianza para operar sin polear Bitacora.

*Evidencia:* grep notif/alert/webhook en src/lib/command = vacio; src/lib/command/state.ts:4-14 (verified/expired inalcanzables)

### Rol unico de admin, sin asientos de equipo ni separacion proponente/aprobador (M) `[high-value-gap]`
Cada /command y ruta API da 404 salvo que el email este en BUILT_IN_ADMINS o ADMIN_EMAILS, y todo admitido recibe el verb set completo incluido execute y rollback; la misma persona puede proponer→aprobar→ejecutar. Peor, dar acceso a Command tambien da /admin y /api/migrate. Sumar un asiento hoy requiere cambiar ADMIN_EMAILS en Coolify + restart y una fila en workspace_members (sin ella workspaceIds queda vacio y Cuentas no lista nada). Una tabla de roles + membership por workspace es el fix estructural antes de que un no-dueno o cliente lo use.

*Evidencia:* src/lib/admin.ts:11-23; src/lib/command/access.ts:19-28; command/layout.tsx:17-22; blueprint/route.ts:40-41; Coolify env (ADMIN_EMAILS ausente)

### Reporte a cliente solo del motor Google; el ledger before/after no se exporta (M) `[high-value-gap]`
El unico reporte a cliente es /r/[account]/[token], proxy del HTML firmado del sentinel externo — solo Google y es performance del motor, no actividad del Command Center. No hay reporte Meta alguno, y la Bitacora (justo el diferenciador vs UIs nativas) no tiene export CSV/PDF ni resumen semanal de 'que cambiamos'. El paso 'reportar al cliente' termina fuera del producto para Meta y a medias para Google.

*Evidencia:* src/app/r/[account]/[token]/page.tsx:1-13; src/app/command/bitacora/bitacora-client.tsx

### Google no-SEARCH sin soporte en create/edit: PMax, Display, Shopping (L) `[high-value-gap]`
readTree() lanza en cualquier canal no-SEARCH y el blueprint de create esta hardcodeado a channel:'SEARCH', asi que buena parte de las cuentas Google reales (sobre todo Performance Max, donde vive mucho del gasto) no se pueden crear ni editar en el workbench — solo los verbs v1 de optimize (budget/pause/enable/negatives) las tocan. Esto topa la 'operacion full lifecycle' a un tipo de canal por red; cada canal nuevo es un schema+compile path nuevo sobre el mismo rail.

*Evidencia:* src/lib/command/blueprint/compile.ts:82; v2.3 design §a/§h; v2 design §4

### Meta (con credenciales) solo OUTCOME_TRAFFIC, link ads con URL de imagen, sin edit mode (L) `[high-value-gap]`
El builder Meta hardcodea objective OUTCOME_TRAFFIC/LINK_CLICKS/LOWEST_COST y targeting pais+edad; OUTCOME_LEADS/OUTCOME_SALES (promoted_object, pixel), subida binaria de imagenes (/adimages, image_hash) y el edit mode Meta entero estan diferidos (startEdit early-return salvo google_ads). Lead-gen/conversion y creativos subidos corren la mayoria de cuentas gestionadas, y una campana creada aqui queda inoperable en la misma herramienta un dia despues. El spec dice que la maquinaria transfiere 1:1 con 'un promoted_object mas' + pickers. Gated tras activar credenciales Meta (§4).

*Evidencia:* src/lib/command/networks/meta.ts:151-216; v2.2 design §a + Explicitly deferred; cuentas-client.tsx:98-99 (startEdit google-only)

### Onboarding de cuentas Meta es config de dev, sin OAuth ni conexiones por workspace (L) `[high-value-gap]`
Las cuentas Meta salen exclusivamente de env (META_AD_ACCOUNT_IDS/META_SYSTEM_USER_TOKEN/META_PAGE_ID/META_APP_SECRET) y /conexiones solo renderiza una tarjeta read-only derivada de process.env; no hay flujo OAuth, storage de credenciales Meta por workspace, ni forma de sumar la cuenta de un cliente sin redeploy, y cada ad publica bajo un unico META_PAGE_ID de env. Google en cambio ya tiene connect real con refresh token por conexion. Es prerequisito de la operacion multi-cliente en el lado Meta de la vision.

*Evidencia:* src/lib/command/networks/meta.ts:30-33; src/app/conexiones/page.tsx:103-110; v2.2 design §e


## 3. Deuda diferida a proposito

### CC_DRY_RUN no rehearsa blueprints multi-accion (falla cerrado; create se ensaya con campana PAUSED desechable) `[deferred-by-design]`
Dry-run no devuelve resourceNames, asi que el plan runner falla cerrado en el primer tmp: ref; solo acciones v1 sueltas se rehearsan. Documentado como caveat permanente, no como bug.

*Evidencia:* DEPLOY-NOTES CC_DRY_RUN and multi-action blueprints; progress.md v2 Minor #3

### Copiloto AI v2.4/v2.5 sin construir; suggest es per-field y Google-only `[deferred-by-design]`
propose_blueprint_patch, applyBlueprintPatch, ProvenanceMap (dato/ia/auto), el switch Manual/AI por paso y chat como modo de entrada son el proximo tramo del roadmap y siguen sin construir. Hoy la asistencia AI es solo suggest per-field, Google-only (sin Meta AI) — es la mayor tranche planeada de la vision 'AI-assisted at every step'.

*Evidencia:* v2 design §13 (v2.4/v2.5); v2.2 spec Explicitly deferred (suggest Meta)

### Sin transacciones DB en ops multi-write (compile delete-then-insert, approve dos writes) `[deferred-by-design]`
El codebase no usa transacciones en ningun lado; documentado como acotado/self-healing (insert batched atomico, status al final, retry recupera) pero listado como address-before-broad-exposure.

*Evidencia:* DEPLOY-NOTES v2 Known limitations bullet 1; progress.md Task 10

### Gate preview determinista; validateOnly/validate_only solo corre en publish `[deferred-by-design]`
El review puede pasar 'gates OK' y aun bloquearse en publish (pais no mapeado, sin conversion action ENABLED). Surfaceado como validateOnlyDeferred:true; un endpoint de rehearsal pre-publish lo apretaria.

*Evidencia:* DEPLOY-NOTES v2 Known limitations bullet 2; preview.ts:23,66; revisar-client.tsx:12

### Staleness estructural en edit: DRIFT solo status+budget, creates expected=null, sin resume, ventana de doble-serving en RSA-replace `[deferred-by-design]`
add_negatives/create_keywords/create_ad llevan expected=null y auto-pasan, protegidos solo por validateOnly + stop-on-first-failure + TTL 60min; un edit fallido no reanuda (rollback-o-dejar). Todo surfaceado en UI; resume endpoint y drift estructural son los closers futuros.

*Evidencia:* v2.3 design §d + Top risks + §h

### Edit slice-1: sin nuevos subarboles de ad-group (v2.3.1), reload descarta ediciones, sin bulk edit, sin Meta edit `[deferred-by-design]`
La creacion de subarbol de ad-group se empuja a v2.3.1 (para sacar tmp: ref del edit), reload descarta pending con confirm, y bulk edit multi-campana + Meta edit son specs futuras sin scope.

*Evidencia:* v2.3 design §h Explicitly Deferred bullets 1 y ultimo

### Meta slice-1 pins: 1 adset, solo ABO, enum de 6 paises, sin targeting detallado, special_ad_categories=[] `[deferred-by-design]`
Un adset por blueprint, budgets ABO (no CBO), enum fijo MX/US/AR/CO/CL/PE (sin EU/DSA), sin intereses/audiencias/placements/idiomas/genero, sin bid caps, y special_ad_categories=[] hardcodeado (riesgo de policy si el target es HOUSING/EMPLOYMENT/CREDIT/FINANCIAL, checklist item 6). Cada uno es un corte YAGNI con path de vuelta.

*Evidencia:* v2.2 design Slice-1 scope decisions + Explicitly deferred

### Assets/audiencias/extensiones sin soporte en ambas redes; cross-account templates y policy pre-check son v2.5+ `[deferred-by-design]`
Sitelinks, callouts, biblioteca de imagenes/assets, audiencias y remarketing quedan fuera de create y edit en ambas redes; una Search moderna sin sitelinks/assets queda handicapeada, asi que el operador termina el setup en la UI nativa.

*Evidencia:* v2 design §1 Explicitly OUT + §13; v2.3 §h

### Asimetrias por diseno: creates solo via blueprint, enable de Google es manual, verbs internos siguen internos `[deferred-by-design]`
La ruta manual v1 rechaza create_* (creates solo por blueprint), las campanas nacen PAUSED sin auto-enable (el operador enciende en Acciones o UI nativa), y remove_entity/remove_negatives siguen internal-only. Invariantes documentadas; cambiarlas es decision de diseno, no bugfix.

*Evidencia:* DEPLOY-NOTES v2 Known limitations bullet 3; v2.3 §a; progress.md Task 14

### Sin acciones programadas/por tiempo (dayparting, enable diferido, budget scheduling) `[deferred-by-design]`
Todo ejecuta al instante en el click de Ejecutar; no hay cron ni run_at ni dayparting en src/lib/command. Subir budget viernes/bajar lunes, encender en fecha de launch o pausar en feriado no se pueden armar por adelantado. Un run_at en cc_actions + un cron reusando executeAction lo cierra.

*Evidencia:* grep cron/schedule en src/lib/command y /api/command = vacio; executor.ts (execute sincrono request-scoped)

### Refactors internos + cosmeticos + test-coverage triaged (no bloqueantes) `[polish]`
Dispatcher de compile compartido diferido (branch inline en 4 sitios) y budget_update Meta mantiene Math.round en vez de microsToCents estricto; cosmeticos: sin status rolled_back (un revertido lee 'executed'), rollback gate-block = 502 generico, botones hand-rolled, layout desktop-only (tablas con scroll horizontal en movil), TARGET_ROAS inalcanzable desde UI. Tests fail-closed pendientes (create_budget en cap, create_campaign sin status, Meta validate() atrapando throws, requirePageId). Barato de cerrar, guarda comportamiento fail-closed que el diseno asume.

*Evidencia:* v2.2 design §h; progress.md MINORs Task 6/11/13, v2.2/v2.3; ui-kit.tsx:313
